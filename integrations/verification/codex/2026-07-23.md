## Verdict

**failed** — but the cause is on the UDS side, not Codex's.

## What actually happened

Codex read the project, listed all 65 files under `.standards/`, read `AGENTS.md`, and gave a
correct answer that cleanly separated what it could verify (Prisma in `package.json`) from what
it could not (no schema, no connection config, no migration script).

**It reasoned exactly as the standard asks. It just never used the declared form.**

## Why: the file it was given does not contain the rule

| File | Size | Contains `[Confirmed]` / Certainty Classification? |
|------|-----:|---------------------------------------------------|
| `integrations/codex/AGENTS.md` — the integration in this repo | 6,457 B | **yes** (`### 3. Certainty Classification`) |
| `AGENTS.md` as produced by `uds init` | 5,594 B | **no** |

`uds init` generates a *universal summary* AGENTS.md that indexes `.standards/` by filename:

```
- `.standards/anti-hallucination.ai.yaml` — anti-hallucination
...
- Read `.standards/` files for complete rules before making changes
```

The four-tag vocabulary lives inside `.standards/anti-hallucination.ai.yaml`. Codex saw the
index, did not open the file, and therefore never met the rule. `--content-mode full` does not
change this: it produces a byte-identical AGENTS.md.

## The part that matters beyond this run

`scripts/check-ai-agent-sync.sh` greps `integrations/codex/AGENTS.md` for rule keywords and
passes. **That is not the file an adopter receives.** The check validates one artifact while
`uds init` ships a different one, so a rule can be present for the checker and absent for
every user, with nothing going red.

This is the same shape as XSPEC-355 OQ5/OQ6 and XSPEC-357 §1 — a claim whose verification
looks at something other than what it claims about — but it is the first instance where the
gap is between **what we check** and **what we ship**.
