## Workflow | 工作流程

> 📖 以下為流程的**參考結構**，非可執行步驟；自動化／強制執行請用 adoption layer 工具鏈（XSPEC-095）。

## Review Categories | 審查類別

1. **Functionality** - Does it work correctly? | 功能是否正確？
2. **Design** - Is the architecture appropriate? | 架構是否合適？
3. **Quality** - Is the code clean and maintainable? | 程式碼是否乾淨可維護？
4. **Readability** - Is it easy to understand? | 是否容易理解？
5. **Tests** - Is there adequate test coverage? | 測試覆蓋是否足夠？
6. **Security** - Are there any vulnerabilities? | 是否有安全漏洞？
7. **Performance** - Is it efficient? | 是否有效率？
8. **Error Handling** - Are errors handled properly? | 錯誤處理是否妥當？

- **BLOCKING** [src/checkout.js:4](src/checkout.js:4) — Production database credentials are hard-coded in source. Rotate this exposed password and load credentials from a secret manager/environment configuration.

- **BLOCKING** [src/checkout.js:13](src/checkout.js:13) — Checkout accepts unvalidated client-controlled quantities. Negative quantities increase inventory and can produce a negative order total; fractional, zero, non-numeric, and excessively large values are also accepted.

- **BLOCKING** [src/checkout.js:14](src/checkout.js:14) — A missing product makes `p` undefined and throws at `p.price`, after any earlier cart items may already have had inventory changed.

- **BLOCKING** [src/checkout.js:14](src/checkout.js:17) — The stock value is read but never enforced. The update can drive stock below zero, and the separate read/update permits concurrent checkouts to oversell.

- **BLOCKING** [src/checkout.js:13](src/checkout.js:24) — Inventory updates and order creation are not transactional. A failure partway through leaves partial stock deductions; an order-insert failure leaves deducted stock without an order.

- **BLOCKING** [src/checkout.js:24](src/checkout.js:31) — The endpoint reports a successful order before payment has completed. The `fetch` promise is neither awaited nor checked, so rejected/failed charges still return success and leave an unpaid order.

- **IMPORTANT** [src/checkout.js:26](src/checkout.js:26) — The payment request has no error handling, timeout/idempotency protection, or response-status validation. Retrying after an uncertain network failure can double-charge the order.

- **IMPORTANT** [src/checkout.js:24](src/checkout.js:24) — `req.body.uid` is trusted as the purchaser identity. Unless an upstream authenticated middleware overwrites it, callers can create orders for arbitrary users; use the authenticated principal instead.

- **IMPORTANT** [src/checkout.js:10](src/checkout.js:10) — `req.body.cart` and its items are not validated. Missing/malformed carts throw, and an arbitrarily large cart causes unbounded sequential database work.

- **IMPORTANT** [src/checkout.js:16](src/checkout.js:16) — Monetary totals are calculated with JavaScript floating-point arithmetic and then discounted, risking persisted/charged rounding errors. Use integer minor units or a decimal money type with explicit rounding.

- **IMPORTANT** [src/checkout.js:3](src/checkout.js:3) — A single connection is created during module import with no retry/reconnect handling or shutdown lifecycle. A temporary database outage can prevent startup, and a dropped connection breaks later checkouts. A managed pool is appropriate.

- **IMPORTANT** [src/checkout.js:3](src/checkout.js:7) — The database connection does not require TLS. For a production database, credentials and order data may transit unencrypted unless transport security is enforced elsewhere.

- **IMPORTANT** [src/checkout.js:9](src/checkout.js:9) — No local error handling maps database/payment failures to safe HTTP responses. Express will likely return generic 500s, while prior inventory mutations may already be committed.

- **SUGGESTION** [src/checkout.js:13](src/checkout.js:17) — Each cart item performs a sequential select and update (N+1 round trips). Batch product lookup and use conditional stock updates within a transaction for better checkout latency.

- **SUGGESTION** [package.json:4](package.json:4) — There is no test script or visible test coverage for checkout’s validation, stock contention, rollback, rounding, and payment-failure paths.
