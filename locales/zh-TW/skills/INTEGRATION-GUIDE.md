# 技能整合指南

本指南說明如何在開發工作流程中整合使用多個技能。

## 概覽

通用開發標準提供 9 個 Claude Code 技能，共同支援完整的開發生命週期：

```
規劃 → 編碼 → 測試 → 審查 → 提交 → 發布
  │      │      │      │      │      │
  ▼      ▼      ▼      ▼      ▼      ▼
需求   專案   測試   程式碼  提交   發布
助手   結構   指南   審查   標準   標準
```

## 技能互動矩陣

| 來源技能 → | 目標技能 | 互動方式 |
|------------|----------|----------|
| 需求助手 | 全部 | 需求影響所有決策 |
| 專案結構 | 測試指南 | 測試位置遵循結構 |
| 測試指南 | 程式碼審查 | 審查檢查測試覆蓋 |
| 程式碼審查 | 提交標準 | 審查通過後才能提交 |
| 提交標準 | 發布標準 | 提交決定版本號 |
| Git 工作流程 | 全部 | 分支策略影響所有 |

## 工作流程場景

### 場景 1：新功能開發

```mermaid
graph LR
    A[需求] --> B[專案結構]
    B --> C[撰寫程式碼]
    C --> D[測試指南]
    D --> E[程式碼審查]
    E --> F[提交標準]
    F --> G[Git 工作流程]
```

**使用的技能**：
1. **需求助手** - 釐清功能需求
2. **專案結構** - 決定檔案位置
3. **測試指南** - 為新功能撰寫測試
4. **程式碼審查** - 取得程式碼審查
5. **提交標準** - 撰寫正確的提交訊息
6. **Git 工作流程** - 透過正確的分支策略合併

### 場景 2：錯誤修正

```mermaid
graph LR
    A[識別錯誤] --> B[撰寫測試]
    B --> C[修正程式碼]
    C --> D[審查]
    D --> E[提交]
```

**使用的技能**：
1. **測試指南** - 先撰寫失敗的測試
2. **程式碼審查** - 快速審查修正
3. **提交標準** - 使用 `fix:` 類型
4. **Git 工作流程** - 建立 `fix/` 分支

### 場景 3：發布準備

```mermaid
graph LR
    A[彙整變更] --> B[更新版本]
    B --> C[更新變更日誌]
    C --> D[建立標籤]
    D --> E[發布]
```

**使用的技能**：
1. **提交標準** - 審查提交歷史
2. **發布標準** - 決定版本升級
3. **文件指南** - 更新變更日誌
4. **Git 工作流程** - 標籤與發布

## 技能優先順序解析

當多個技能適用時，使用此優先順序：

| 優先 | 類別 | 技能 |
|------|------|------|
| 1 | 安全 | AI 協作（防幻覺） |
| 2 | 正確性 | 測試指南、程式碼審查 |
| 3 | 流程 | Git 工作流程、提交標準 |
| 4 | 文件 | 文件指南、發布標準 |
| 5 | 指引 | 需求助手、專案結構 |

## 常見技能組合

### 程式碼品質組合

同時使用以獲得高品質程式碼：
- **測試指南** + **程式碼審查**
- 確保測試存在且程式碼已審查

### 發布組合

同時使用以正確發布：
- **提交標準** + **發布標準** + **Git 工作流程**
- 確保正確的版本控制和變更日誌

### 新專案組合

啟動專案時同時使用：
- **專案結構** + **文件指南** + **Git 工作流程**
- 建立正確的基礎

## 衝突解決

### 測試 vs 速度

**衝突**：需要快速交付但測試指南要求覆蓋率

**解決方式**：
1. 撰寫關鍵路徑測試（測試指南）
2. 記錄技術債（文件指南）
3. 使用適當的提交類型（提交標準）

### 審查 vs 緊急

**衝突**：緊急修正需要立即部署

**解決方式**：
1. 使用加速審查（程式碼審查 - 緊急流程）
2. 建立 hotfix 分支（Git 工作流程）
3. 記錄例外（文件指南）

## 最佳實踐

### 1. 從需求開始

在使用其他技能之前，始終從明確的需求開始：
```
使用者故事 → 驗收標準 → 實作 → 測試 → 審查
```

### 2. 提早測試

在程式碼審查之前調用測試指南：
```
撰寫程式碼 → 撰寫測試 → 自我審查 → 請求審查
```

### 3. 一致的提交

全程使用提交標準，而非只在最後：
```
每個邏輯變更 = 一個具有正確格式的提交
```

### 4. 記錄決策

使用文件指南記錄重要決策：
```
做出決策 → 記錄在 ADR → 在程式碼中引用
```
