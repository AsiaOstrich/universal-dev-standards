# AI 回應導航標準

> **語言**: [English](../../../core/ai-response-navigation.md) | 繁體中文 | [简体中文](../../zh-CN/core/ai-response-navigation.md)

**版本**: 1.0.0
**最後更新**: 2026-03-25
**適用範圍**: 所有使用 AI 輔助開發的專案
**範圍**: universal
**產業標準**: 無（新興 AI 工具實踐）
**參考**: [SPEC-STD-08](../../../docs/specs/standards/SPEC-STD-08-ai-response-navigation.md)

---

## 目的

本標準定義 AI 回應的導航行為：每個實質性的 AI 回應必須包含情境化的下一步建議與推薦選項。確保使用者在開發工作流程中持續被引導，無需記憶所有可用命令。

**問題**：使用者在收到 AI 回應後不知道下一步該做什麼。面對 30+ 個斜線命令，認知負擔高且工作流程連續性中斷。

**解決方案**：在每個實質性 AI 回應結尾附加標準化的「導航區塊」，包含情境模板、推薦標記和彈性選項數量。

---

## 核心規則

### 規則 1：每個實質性回應都包含導航

每個構成「邏輯回應單元」的 AI 回應必須以導航區塊作為結尾。

**以下情況視為邏輯回應單元：**

1. 完成了一項任務或子任務
2. 提供了分析、解釋或建議
3. 向使用者詢問問題或要求選擇
4. 報告了錯誤或異常狀態
5. 展示了程式碼變更結果

**豁免**：極短確認回覆（如「好的」「已完成」「了解」）不構成獨立的邏輯單元，可以省略導航區塊。

### 規則 2：推薦並說明理由

提供多個選項時，使用 ⭐ **推薦** 標記最佳選項，並在 ` — ` 之後附上理由。

| 情境 | 標記方式 |
|------|----------|
| 有一個選項明顯較優 | ⭐ **推薦** — [理由] |
| 各選項無明顯優劣 | 說明各選項適用場景，不標記 ⭐ |
| 只有一個合理的下一步 | 直接建議，不需要 ⭐ |

### 規則 3：匹配情境

根據回應類型使用對應的模板（參見[情境模板](#情境模板)）。

### 規則 4：彈性數量

根據情境複雜度調整選項數量：

| 情境 | 選項數量 | 理由 |
|------|----------|------|
| 任務完成 | 2–3 | 建議後續工作流程步驟 |
| 使用者詢問 | 2–5 | 依問題複雜度調整 |
| 錯誤/失敗 | 1–3 | 聚焦修復方向 |
| 進行中 | 1–2 | 繼續或調整 |
| 資訊回覆 | 1–3 | 探索方向 |

**硬性上限**：不得超過 5 個選項。裁減至最相關的選項。

### 規則 5：優先使用斜線命令

當下一步建議對應到已知的斜線命令時，使用 `` `/command` `` 格式引用，讓使用者可以直接複製執行。沒有對應命令的步驟使用自然語言描述。

## 情境模板

### 模板 1：任務完成

任務、技能執行或程式碼修改完成時使用。

```markdown
> **建議下一步：**
> - 執行 `/command1` 做 X
> - 執行 `/command2` 做 Y ⭐ **推薦** — [理由]
> - 執行 `/command3` 做 Z
```

### 模板 2：使用者詢問

AI 需要使用者做出選擇或提供資訊時使用。

```markdown
> **請選擇：**
> - **(A) 選項描述** — 補充說明
> - **(B) 選項描述** ⭐ **推薦** — [理由]
> - **(C) 選項描述** — 補充說明
```

### 模板 3：錯誤/失敗

報告錯誤、失敗或異常狀態時使用。

```markdown
> **建議修復方向：**
> - 選項描述 ⭐ **推薦** — [理由]
> - 選項描述
```

### 模板 4：進行中

完成多步驟任務的中間階段時使用。

```markdown
> **目前進度：[N/M]。下一步：**
> - 繼續下一階段 ⭐ **推薦**
> - 調整方向或參數
```

### 模板 5：資訊回覆

回答知識性問題或提供說明時使用。

```markdown
> **建議下一步：**
> - 深入了解 [相關主題]
> - 執行 `/command` 將知識應用到實作
```

## 格式規範

### 結構

所有導航區塊使用 Markdown 引用區塊（`>`）：

```markdown
> **[標題]：**
> - [選項/建議] ⭐ **推薦** — [理由]
> - [選項/建議] — [補充說明]
```

### 依情境的標題

| 情境 | 標題 |
|------|------|
| 任務完成 | `建議下一步` |
| 使用者詢問 | `請選擇` |
| 錯誤/失敗 | `建議修復方向` |
| 進行中 | `目前進度：[N/M]。下一步` |
| 資訊回覆 | `建議下一步` |

### 本地化

對於雙語專案，導航區塊遵循對話語言。本地化對照：

| English | 繁體中文 | 简体中文 |
|---------|----------|----------|
| Suggested next steps | 建議下一步 | 建议下一步 |
| Please choose | 請選擇 | 请选择 |
| Suggested resolution | 建議修復方向 | 建议修复方向 |
| Progress: [N/M]. Next | 目前進度：[N/M]。下一步 | 当前进度：[N/M]。下一步 |
| Recommended | 推薦 | 推荐 |

## 與技能的整合

### 現有技能

已有 `## Next Steps Guidance | 下一步引導` 區塊的技能：

1. **保留**技能特有的場景化建議內容
2. **對齊**格式以符合本標準定義的模板
3. **加入**推薦標記（⭐）

### 新技能

新技能應包含遵循本標準模板的 `## Next Steps Guidance | 下一步引導` 區塊。

### 超越技能

本標準適用於**所有 AI 回應**，不僅限於技能執行。一般對話、程式碼審查、除錯過程及任何其他互動，在產生實質性回應時都必須遵循這些導航規則。

## 跨 AI 工具適用性

本標準與工具無關。適用於所有使用 UDS 標準的 AI 工具：

- Claude Code
- Cursor
- Windsurf
- GitHub Copilot
- Cline
- Roo Code
- Augment Code
- Trae
- 其他相容工具

每個工具的整合層負責根據其 UI 能力渲染導航區塊，但行為規則保持一致。

## 範例

### 範例 1：完成功能實作後

```markdown
> **建議下一步：**
> - 執行 `/test` 為新功能撰寫測試
> - 執行 `/commit` 提交變更 ⭐ **推薦** — 變更已完整且通過驗證
> - 執行 `/review` 進行自我審查
```

### 範例 2：詢問使用者設計問題

```markdown
> **請選擇：**
> - **(A) 使用新的資料庫表** — 更靈活，遷移成本較高
> - **(B) 擴展現有表增加欄位** ⭐ **推薦** — 遷移更簡單，滿足當前需求
> - **(C) 使用 NoSQL 儲存** — 適合非結構化資料但增加基礎設施
```

### 範例 3：報告測試失敗

```markdown
> **建議修復方向：**
> - 檢查 `tests/auth.test.js:42` 的失敗斷言 ⭐ **推薦** — 錯誤訊息指向空引用
> - 執行 `/debug` 進行系統性除錯
> - 還原上次變更並調查
```

### 範例 4：工作流程中間進度更新

```markdown
> **目前進度：[2/5]。下一步：**
> - 繼續第 3 階段：實作 ⭐ **推薦**
> - 在繼續前重新檢視第 2 階段需求
```

### 範例 5：回答技術問題

```markdown
> **建議下一步：**
> - 探索 `config/auth.yaml` 中的相關配置
> - 執行 `/sdd` 建立規格（若計畫修改此行為）
```
