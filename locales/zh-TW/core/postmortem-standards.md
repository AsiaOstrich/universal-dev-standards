# Postmortem 標準（無責事後檢討）

> 版本: 1.0.0 | 最後更新: 2026-04-01

## 概述

本文件定義生產環境事件後進行無責 Postmortem 的標準。Postmortem 是一個結構化流程，用於分析發生了什麼、為什麼發生、以及如何防止再次發生 — 不歸咎於個人。

事件回應工作流程（偵測 → 分級 → 緩解 → 解決）請參閱 [Incident Response Assistant](../../../skills/incident-response-assistant/SKILL.md)。

---

## 觸發條件

### 強制執行

| 條件 | 理由 |
|------|------|
| **SEV-1 事件**（一律） | 任何完全服務中斷需要徹底分析 |
| **SEV-2 事件持續 > 1 小時** | 延長的部分中斷指出系統性問題 |
| **任何涉及資料遺失的事件** | 資料遺失具有不可逆後果 |
| **任何造成 SLA 違反的事件** | SLA 違規有合約影響 |
| **任何涉及安全漏洞的事件** | 安全事件需要詳細鑑識分析 |

### 建議執行

| 條件 | 理由 |
|------|------|
| SEV-2 事件持續 < 1 小時 | 短暫但重大；可能揭示系統性風險 |
| SEV-3 重複事件（30 天內 2 次以上，相同根因） | 重複表示未解決的系統性問題 |
| 具有大潛在影響的 Near-miss | 無實際損害的學習機會 |

## 無責原則

| 原則 | 說明 |
|------|------|
| **人會犯錯** | 錯誤是複雜系統的自然產物，不是個人缺陷 |
| **聚焦系統** | 分析系統如何允許錯誤發生，而非誰犯了錯 |
| **心理安全** | 參與者可以公開分享，不必擔心懲罰 |
| **學習導向** | 目標是學習和改善，不是追究責任 |

### 語言指南

| 正確（系統聚焦） | 錯誤（歸咎聚焦） |
|-----------------|-----------------|
| 「部署包含未覆蓋的邊界案例」 | 「工程師 A 部署了有 Bug 的程式碼」 |
| 「監控系統未在 5 分鐘內偵測到異常」 | 「值班工程師忽略了告警」 |

## 根因分析方法

| 方法 | 最適用情境 |
|------|-----------|
| **5 Whys** | 單一事件鏈，因果關係清楚 |
| **Ishikawa（魚骨圖）** | 多個可能原因，需要分類 |
| **Fault Tree Analysis** | 安全關鍵系統、機率分析 |
| **Timeline Analysis** | 涉及多團隊、多系統的複雜事件 |
| **Change Analysis** | 在近期變更後發生的事件 |

### Root Cause vs Trigger vs Contributing Factor

| 類型 | 定義 | 範例 |
|------|------|------|
| **Root Cause** | 若移除，事件不會發生 | CI/CD 缺乏 SQL 效能分析 |
| **Trigger** | 直接啟動事件的事件 | 新功能部署 |
| **Contributing Factor** | 加重影響但非根因 | 監控告警延遲 10 分鐘 |

## Quick Reference Card

### 是否需要撰寫 Postmortem？
```
SEV-1？                   → 是（一律）
SEV-2 且 > 1 小時？       → 是
資料遺失？                → 是
SLA 違反？                → 是
安全漏洞？                → 是
SEV-2 且 < 1 小時？       → 建議
SEV-3 重複發生？          → 建議（簡化版）
SEV-3/4 首次發生？        → 否（記錄於事件日誌）
```

### Postmortem 時間線
```
事件解決 → 24h → 收集事實
         → 48h → 撰寫草稿
         → +3 個工作日 → 召開會議
         → +24h → 定稿並發布
         → 持續 → 追蹤行動項目
```
