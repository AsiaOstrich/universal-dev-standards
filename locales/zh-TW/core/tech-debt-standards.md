# 技術債管理標準

> 版本: 1.0.0 | 最後更新: 2026-04-01

## 概述

技術債代表因選擇權宜之計而非更佳方案所產生的未來重工隱含成本。本標準定義如何分類、登記、排序、預算、衡量及追蹤技術債。

## 1. 技術債類型

| 類型 | 說明 | 範例 |
|------|------|------|
| **Design Debt** | 架構捷徑或次優設計決策 | 缺少抽象層、緊耦合 |
| **Code Debt** | 增加維護負擔的程式碼品質問題 | 重複程式碼、過長方法、Magic Number |
| **Test Debt** | 不足或低品質的測試覆蓋 | 缺少單元測試、不穩定測試 |
| **Documentation Debt** | 缺失、過時或不正確的文件 | 未文件化的 API、過時 README |
| **Dependency Debt** | 過時、有漏洞或不必要的依賴 | 未修補的函式庫、已棄用框架 |
| **Infrastructure Debt** | 次優的建置、部署或維運工具 | 手動部署、缺少監控 |

### 蓄意 vs 非蓄意

- **蓄意（Deliberate）**：有意識地選擇捷徑，應立即登記
- **非蓄意（Inadvertent）**：因知識不足而產生，發現後應立即登記

## 2. 技術債登記表

每個技術債項目必須記錄以下 11 個欄位：

| # | 欄位 | 說明 |
|---|------|------|
| 1 | **ID** | 唯一識別碼，格式 `TD-NNN` |
| 2 | **Title** | 簡短描述性標題 |
| 3 | **Type** | 上述 6 種類型之一 |
| 4 | **Source** | 引入方式（蓄意/非蓄意） |
| 5 | **Impact** | 業務與技術影響描述 |
| 6 | **Estimated Cost** | 解決所需工作量 |
| 7 | **Interest** | 不解決的持續成本 |
| 8 | **Priority** | 優先等級（P0-P3） |
| 9 | **Owner** | 負責團隊或個人 |
| 10 | **Created Date** | 登記日期 |
| 11 | **Target Resolution Date** | 計劃解決日期 |

## 3. 預算分配

| 團隊狀態 | 預算 | 說明 |
|---------|------|------|
| 新專案（< 6 個月） | **10%** | 預防性維護 |
| 成熟專案（6+ 個月） | **15%** | 持續維護 |
| 高債務專案 | **20-30%** | 積極償還 |

## 4. 優先排序矩陣

|  | 低工作量 | 中工作量 | 高工作量 |
|--|---------|---------|---------|
| **高影響** | P0 — 立即 | P1 — 下個 Sprint | P1 — 下個 Sprint |
| **中影響** | P1 — 下個 Sprint | P2 — 本季 | P2 — 本季 |
| **低影響** | P2 — 本季 | P3 — Backlog | P3 — Backlog |

### 利息類型

| 利息類型 | 說明 |
|---------|------|
| **時間利息** | 繞過債務所需的額外開發時間 |
| **風險利息** | Bug、當機或安全事件的機率增加 |
| **人才利息** | 對開發者體驗與留才的負面影響 |

## 5. 量化指標

| 指標 | 公式 | 目標 |
|------|------|------|
| 總債務量 | 所有未關閉債務項目數 | 下降趨勢 |
| 債務比率 | 債務項目 / 總 Backlog × 100% | < 15% |
| 平均存活天數 | 總天數 / 債務項目數 | < 90 天 |
| 類型分布 | 各類型佔比 | 單一類型不超過 40% |
| 高優先比率 | (P0+P1) / 總數 × 100% | < 20% |

## 6. Commit 標記

引入債務時：`Tech-Debt: TD-042 introduced`
解決債務時：`Tech-Debt: TD-042 resolved`
