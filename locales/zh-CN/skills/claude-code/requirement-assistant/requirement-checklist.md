# 需求完整性检查清单

> **语言**: [English](../../../../../skills/claude-code/requirement-assistant/requirement-checklist.md) | 繁体中文

**版本**: 1.0.0
**最后更新**: 2025-12-24
**适用范围**: Claude Code Skills

---

## 目的

本文件提供用于验证需求完整性和品质的检查清单。

---

## 快速检查清单

在提交任何需求之前，请验证以下要点:

| 项目 | 勾选 | 问题 |
|------|:-----:|----------|
| 问题 | [ ] | 我们要解决什么问题？ |
| 使用者 | [ ] | 谁会从中受益？ |
| 价值 | [ ] | 为什么这很重要？ |
| 范围 | [ ] | 包含和排除什么？ |
| 标准 | [ ] | 我们如何知道已完成？ |
| 优先级 | [ ] | 这有多紧急？ |

---

## 按需求类型的详细检查清单

### 功能需求检查清单

#### 1. 问题定义

- [ ] 问题陈述清楚
- [ ] 识别目前的痛点
- [ ] 量化影响（如可能）
- [ ] 了解根本原因

#### 2. 使用者情境

- [ ] 识别目标使用者/角色
- [ ] 描述使用者目标
- [ ] 记录目前的替代方案
- [ ] 绘制使用者旅程（如复杂）

#### 3. 解决方案描述

- [ ] 描述功能（非实作）
- [ ] 概述使用者互动
- [ ] 陈述预期结果
- [ ] 定义成功指标

#### 4. 验收标准

- [ ] 所有标准都可测试
- [ ] 涵盖正常路径
- [ ] 定义错误情境
- [ ] 考虑边界情况
- [ ] 效能标准（如适用）

#### 5. 范围定义

- [ ] 列出范围内项目
- [ ] 列出范围外项目
- [ ] 注明未来考虑事项
- [ ] 识别依赖关系

#### 6. 优先级与时间表

- [ ] 分配优先级（P0-P3 或 MoSCoW）
- [ ] 提供业务理由
- [ ] 注明截止日期（如有）
- [ ] 识别发布目标

---

### 错误报告检查清单

#### 1. 描述

- [ ] 标题清晰且具体
- [ ] 描述简洁
- [ ] 陈述影响/严重性

#### 2. 重现

- [ ] 步骤已编号
- [ ] 步骤可重现
- [ ] 列出先决条件
- [ ] 提供测试资料（如需要）

#### 3. 行为

- [ ] 描述预期行为
- [ ] 描述实际行为
- [ ] 差异清楚

#### 4. 证据

- [ ] 附加截图（如 UI）
- [ ] 包含错误日志（如可用）
- [ ] 提供堆叠追踪（如崩溃）

#### 5. 环境

- [ ] 作业系统和版本
- [ ] 浏览器和版本（如网页）
- [ ] 应用程式版本
- [ ] 相关配置

#### 6. 分类

- [ ] 分配严重性
- [ ] 分配优先级
- [ ] 识别元件/区域
- [ ] 连结相关问题

---

### 技术任务检查清单

#### 1. 情境

- [ ] 为什么需要这个？
- [ ] 它解决什么问题？
- [ ] 什么触发了这个任务？

#### 2. 技术细节

- [ ] 描述方法
- [ ] 列出受影响的元件
- [ ] 注明资料库变更
- [ ] 注明 API 变更

#### 3. 范围

- [ ] 将变更什么？
- [ ] 将不会变更什么？
- [ ] 处理向后相容性

#### 4. 标准

- [ ] 完成定义清楚
- [ ] 定义测试需求
- [ ] 识别文件需求

#### 5. 风险

- [ ] 识别技术风险
- [ ] 注明缓解策略
- [ ] 回滚计划（如部署）

## INVEST 验证

使用此方法验证使用者故事:

### Independent（独立）

- [ ] 可以在没有其他故事的情况下交付
- [ ] 没有阻塞依赖关系
- [ ] 可以自由排定优先级

**如果否**: 考虑与依赖故事合并或重新定义范围。

### Negotiable（可协商）

- [ ] 实作细节未锁定
- [ ] 有技术讨论空间
- [ ] 专注于「做什么」，而非「如何做」

**如果否**: 移除实作细节，专注于结果。

### Valuable（有价值）

- [ ] 为使用者/利害关系人提供价值
- [ ] 解决真实问题
- [ ] 好处清楚

**如果否**: 重新考虑是否应该执行。

### Estimable（可估计）

- [ ] 团队可以估计工作量
- [ ] 了解范围
- [ ] 没有重大未知数

**如果否**: 首先需要探索/研究任务。

### Small（小型）

- [ ] 可以在一个 sprint 内完成
- [ ] 通常 1-5 天的工作量
- [ ] 单一连贯的工作部分

**如果否**: 拆分成更小的故事。

### Testable（可测试）

- [ ] 清楚的验收标准
- [ ] 可以编写自动化测试
- [ ] 完成定义明确

**如果否**: 添加具体、可衡量的标准。

## 验收标准品质检查

对于每个验收标准:

| 品质 | 勾选 | 范例 |
|---------|:-----:|---------|
| 具体 | [ ] | "登入按钮为蓝色 (#007bff)" 而非 "按钮看起来不错" |
| 可衡量 | [ ] | "回应 < 200ms" 而非 "快速回应" |
| 可达成 | [ ] | 技术上可行 |
| 相关 | [ ] | 与需求相关 |
| 可测试 | [ ] | 可以编写测试案例 |

### 常见问题

| 问题 | 不良范例 | 良好范例 |
|-------|-------------|--------------|
| 过于模糊 | "系统运作良好" | "系统在 500ms 内回传 200 OK" |
| 主观 | "UI 使用者友善" | "使用者可以在 3 次点击内完成结帐" |
| 缺少边界情况 | "使用者可以上传档案" | "使用者可以上传 1-10MB 档案；较大档案显示错误" |
| 实作细节 | "使用 Redis 快取" | "仪表板在 1 秒内载入" |

## 最终审查检查清单

在提交开发之前：

### 内容品质

- [ ] 标题清楚且具描述性
- [ ] 描述完整
- [ ] 没有模糊的语言
- [ ] 没有未陈述的假设
- [ ] 技术术语已解释

### 完整性

- [ ] 所有必填栏位已填写
- [ ] 验收标准已定义
- [ ] 范围边界清楚
- [ ] 依赖项已列出
- [ ] 优先级已分配

### 一致性

- [ ] 遵循团队范本
- [ ] 一致的术语
- [ ] 已连结到相关项目
- [ ] 标签正确应用

### 准备开发

- [ ] 团队可以估算工作量
- [ ] 没有阻塞的问题
- [ ] 利害关系人批准（如果需要）
- [ ] 设计资产已准备（如果是 UI）
