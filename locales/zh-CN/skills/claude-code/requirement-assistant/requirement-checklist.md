# 需求完整性检查清单

> **语言**: [English](../../../../../skills/claude-code/requirement-assistant/requirement-checklist.md) | 简体中文

**版本**: 1.0.0
**最後更新**: 2025-12-24
**適用範圍**: Claude Code Skills

---

## 目的

本文件提供用於验证需求完整性和品质的检查清单。

---

## 快速检查清单

在提交任何需求之前，請验证以下要点:

| 项目 | 勾選 | 問題 |
|------|:-----:|----------|
| 問題 | [ ] | 我們要解决什麼問題？ |
| 使用者 | [ ] | 誰会從中受益？ |
| 价值 | [ ] | 为什麼这很重要？ |
| 範圍 | [ ] | 包含和排除什麼？ |
| 标准 | [ ] | 我們如何知道已完成？ |
| 優先级 | [ ] | 这有多緊急？ |

---

## 按需求类型的详细检查清单

### 功能需求检查清单

#### 1. 問題定義

- [ ] 問題陳述清楚
- [ ] 識别目前的痛点
- [ ] 量化影響（如可能）
- [ ] 了解根本原因

#### 2. 使用者情境

- [ ] 識别目標使用者/角色
- [ ] 描述使用者目標
- [ ] 记录目前的替代方案
- [ ] 繪制使用者旅程（如複雜）

#### 3. 解决方案描述

- [ ] 描述功能（非实作）
- [ ] 概述使用者互动
- [ ] 陳述预期結果
- [ ] 定義成功指標

#### 4. 驗收标准

- [ ] 所有标准都可测试
- [ ] 涵蓋正常路徑
- [ ] 定義错误情境
- [ ] 考慮邊界情况
- [ ] 效能标准（如適用）

#### 5. 範圍定義

- [ ] 列出範圍內项目
- [ ] 列出範圍外项目
- [ ] 注明未來考慮事项
- [ ] 識别依賴关系

#### 6. 優先级与时间表

- [ ] 分配優先级（P0-P3 或 MoSCoW）
- [ ] 提供业务理由
- [ ] 注明截止日期（如有）
- [ ] 識别發布目標

---

### 错误报告检查清单

#### 1. 描述

- [ ] 標題清晰且具体
- [ ] 描述簡潔
- [ ] 陳述影響/嚴重性

#### 2. 重現

- [ ] 步骤已编号
- [ ] 步骤可重現
- [ ] 列出先决条件
- [ ] 提供测试数据（如需要）

#### 3. 行为

- [ ] 描述预期行为
- [ ] 描述实际行为
- [ ] 差異清楚

#### 4. 证据

- [ ] 附加截图（如 UI）
- [ ] 包含错误日誌（如可用）
- [ ] 提供堆疊追蹤（如崩潰）

#### 5. 環境

- [ ] 作业系统和版本
- [ ] 浏览器和版本（如网页）
- [ ] 应用程序版本
- [ ] 相关配置

#### 6. 分类

- [ ] 分配嚴重性
- [ ] 分配優先级
- [ ] 識别元件/區域
- [ ] 連結相关問題

---

### 技術任务检查清单

#### 1. 情境

- [ ] 为什麼需要这个？
- [ ] 它解决什麼問題？
- [ ] 什麼觸發了这个任务？

#### 2. 技術細节

- [ ] 描述方法
- [ ] 列出受影響的元件
- [ ] 注明数据庫变更
- [ ] 注明 API 变更

#### 3. 範圍

- [ ] 將变更什麼？
- [ ] 將不会变更什麼？
- [ ] 处理向後相容性

#### 4. 标准

- [ ] 完成定義清楚
- [ ] 定義测试需求
- [ ] 識别文件需求

#### 5. 風險

- [ ] 識别技術風險
- [ ] 注明緩解策略
- [ ] 回滾计画（如部署）

## INVEST 验证

使用此方法验证使用者故事:

### Independent（獨立）

- [ ] 可以在没有其他故事的情况下交付
- [ ] 没有阻塞依賴关系
- [ ] 可以自由排定優先级

**如果否**: 考慮与依賴故事合併或重新定義範圍。

### Negotiable（可協商）

- [ ] 实作細节未鎖定
- [ ] 有技術討論空间
- [ ] 專注於「做什麼」，而非「如何做」

**如果否**: 移除实作細节，專注於結果。

### Valuable（有价值）

- [ ] 为使用者/利害关系人提供价值
- [ ] 解决真实問題
- [ ] 好处清楚

**如果否**: 重新考慮是否应該执行。

### Estimable（可估计）

- [ ] 团队可以估计工作量
- [ ] 了解範圍
- [ ] 没有重大未知數

**如果否**: 首先需要探索/研究任务。

### Small（小型）

- [ ] 可以在一个 sprint 內完成
- [ ] 通常 1-5 天的工作量
- [ ] 单一連貫的工作部分

**如果否**: 拆分成更小的故事。

### Testable（可测试）

- [ ] 清楚的驗收标准
- [ ] 可以编写自动化测试
- [ ] 完成定義明确

**如果否**: 添加具体、可衡量的标准。

## 驗收标准品质检查

对於每个驗收标准:

| 品质 | 勾選 | 範例 |
|---------|:-----:|---------|
| 具体 | [ ] | "登入按鈕为藍色 (#007bff)" 而非 "按鈕看起來不錯" |
| 可衡量 | [ ] | "响应 < 200ms" 而非 "快速响应" |
| 可达成 | [ ] | 技術上可行 |
| 相关 | [ ] | 与需求相关 |
| 可测试 | [ ] | 可以编写测试案例 |

### 常見問題

| 問題 | 不良範例 | 良好範例 |
|-------|-------------|--------------|
| 過於模糊 | "系统运作良好" | "系统在 500ms 內回传 200 OK" |
| 主觀 | "UI 使用者友善" | "使用者可以在 3 次点擊內完成結帳" |
| 缺少邊界情况 | "使用者可以上传文件" | "使用者可以上传 1-10MB 文件；較大文件顯示错误" |
| 实作細节 | "使用 Redis 快取" | "儀表板在 1 秒內载入" |

## 最終审查检查清单

在提交开发之前：

### 内容品质

- [ ] 標題清楚且具描述性
- [ ] 描述完整
- [ ] 没有模糊的语言
- [ ] 没有未陳述的假设
- [ ] 技術術語已解釋

### 完整性

- [ ] 所有必填欄位已填写
- [ ] 驗收标准已定義
- [ ] 範圍邊界清楚
- [ ] 依賴项已列出
- [ ] 優先级已分配

### 一致性

- [ ] 遵循团队範本
- [ ] 一致的術語
- [ ] 已連結到相关项目
- [ ] 标签正确应用

### 准备开发

- [ ] 团队可以估算工作量
- [ ] 没有阻塞的問題
- [ ] 利害关系人批准（如果需要）
- [ ] 设计資产已准备（如果是 UI）
