# 技能集成指南

本指南说明如何在开发工作流程中集成使用多个技能。

## 概览

通用开发标准提供 14 个 Claude Code 技能，共同支援完整的开发生命周期：

```
规划 → 编码 → 测试 → 审查 → 提交 → 发布
  │      │      │      │      │      │
  ▼      ▼      ▼      ▼      ▼      ▼
需求   专案   测试   程序码  提交   发布
助手   结构   指南   审查   标准   标准
```

## 技能互动矩阵

| 来源技能 → | 目标技能 | 互动方式 |
|------------|----------|----------|
| 需求助手 | 全部 | 需求影响所有决策 |
| 专案结构 | 测试指南 | 测试位置遵循结构 |
| 测试指南 | 程序码审查 | 审查检查测试覆盖 |
| 程序码审查 | 提交标准 | 审查通过后才能提交 |
| 提交标准 | 发布标准 | 提交决定版本号 |
| Git 工作流程 | 全部 | 分支策略影响所有 |

## 工作流程场景

### 场景 1：新功能开发

```mermaid
graph LR
    A[需求] --> B[专案结构]
    B --> C[撰写程序码]
    C --> D[测试指南]
    D --> E[程序码审查]
    E --> F[提交标准]
    F --> G[Git 工作流程]
```

**使用的技能**：
1. **需求助手** - 厘清功能需求
2. **专案结构** - 决定档案位置
3. **测试指南** - 为新功能撰写测试
4. **程序码审查** - 取得程序码审查
5. **提交标准** - 撰写正确的提交讯息
6. **Git 工作流程** - 透过正确的分支策略合并

### 场景 2：错误修正

```mermaid
graph LR
    A[识别错误] --> B[撰写测试]
    B --> C[修正程序码]
    C --> D[审查]
    D --> E[提交]
```

**使用的技能**：
1. **测试指南** - 先撰写失败的测试
2. **程序码审查** - 快速审查修正
3. **提交标准** - 使用 `fix:` 类型
4. **Git 工作流程** - 建立 `fix/` 分支

### 场景 3：发布准备

```mermaid
graph LR
    A[汇整变更] --> B[更新版本]
    B --> C[更新变更日志]
    C --> D[建立标签]
    D --> E[发布]
```

**使用的技能**：
1. **提交标准** - 审查提交历史
2. **发布标准** - 决定版本升级
3. **文件指南** - 更新变更日志
4. **Git 工作流程** - 标签与发布

## 技能优先顺序解析

当多个技能适用时，使用此优先顺序：

| 优先 | 类别 | 技能 |
|------|------|------|
| 1 | 安全 | AI 协作（防幻觉） |
| 2 | 正确性 | 测试指南、程序码审查 |
| 3 | 流程 | Git 工作流程、提交标准 |
| 4 | 文件 | 文件指南、发布标准 |
| 5 | 指引 | 需求助手、专案结构 |

## 常见技能组合

### 程序码品质组合

同时使用以获得高品质程序码：
- **测试指南** + **程序码审查**
- 确保测试存在且程序码已审查

### 发布组合

同时使用以正确发布：
- **提交标准** + **发布标准** + **Git 工作流程**
- 确保正确的版本控制和变更日志

### 新专案组合

启动专案时同时使用：
- **专案结构** + **文件指南** + **Git 工作流程**
- 建立正确的基础

## 冲突解决

### 测试 vs 速度

**冲突**：需要快速交付但测试指南要求覆盖率

**解决方式**：
1. 撰写关键路径测试（测试指南）
2. 记录技术债（文件指南）
3. 使用适当的提交类型（提交标准）

### 审查 vs 紧急

**冲突**：紧急修正需要立即部署

**解决方式**：
1. 使用加速审查（程序码审查 - 紧急流程）
2. 建立 hotfix 分支（Git 工作流程）
3. 记录例外（文件指南）

## 最佳实践

### 1. 从需求开始

在使用其他技能之前，始终从明确的需求开始：
```
使用者故事 → 验收标准 → 实作 → 测试 → 审查
```

### 2. 提早测试

在程序码审查之前调用测试指南：
```
撰写程序码 → 撰写测试 → 自我审查 → 请求审查
```

### 3. 一致的提交

全程使用提交标准，而非只在最后：
```
每个逻辑变更 = 一个具有正确格式的提交
```

### 4. 记录决策

使用文件指南记录重要决策：
```
做出决策 → 记录在 ADR → 在程序码中引用
```
