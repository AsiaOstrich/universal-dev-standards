# AI 响应导航标准

> **语言**: [English](../../../core/ai-response-navigation.md) | [繁體中文](../../zh-TW/core/ai-response-navigation.md) | 简体中文

**版本**: 1.0.0
**最后更新**: 2026-03-25
**适用范围**: 所有使用 AI 辅助开发的项目
**范围**: universal
**行业标准**: 无（新兴 AI 工具实践）
**参考**: [SPEC-STD-08](../../../docs/specs/standards/SPEC-STD-08-ai-response-navigation.md)

---

## 目的

本标准定义 AI 响应的导航行为：每个实质性的 AI 响应必须包含情境化的下一步建议与推荐选项。确保用户在开发工作流程中持续被引导，无需记忆所有可用命令。

**问题**：用户在收到 AI 响应后不知道下一步该做什么。面对 30+ 个斜杠命令，认知负担高且工作流程连续性中断。

**解决方案**：在每个实质性 AI 响应结尾附加标准化的「导航区块」，包含情境模板、推荐标记和弹性选项数量。

---

## 核心规则

### 规则 1：每个实质性响应都包含导航

每个构成「逻辑响应单元」的 AI 响应必须以导航区块作为结尾。

**以下情况视为逻辑响应单元：**

1. 完成了一项任务或子任务
2. 提供了分析、解释或建议
3. 向用户询问问题或要求选择
4. 报告了错误或异常状态
5. 展示了代码变更结果

**豁免**：极短确认回复（如「好的」「已完成」「了解」）不构成独立的逻辑单元，可以省略导航区块。

### 规则 2：推荐并说明理由

提供多个选项时，使用 ⭐ **推荐** 标记最佳选项，并在 ` — ` 之后附上理由。

| 情境 | 标记方式 |
|------|----------|
| 有一个选项明显较优 | ⭐ **推荐** — [理由] |
| 各选项无明显优劣 | 说明各选项适用场景，不标记 ⭐ |
| 只有一个合理的下一步 | 直接建议，不需要 ⭐ |

### 规则 3：匹配情境

根据响应类型使用对应的模板（参见[情境模板](#情境模板)）。

### 规则 4：弹性数量

根据情境复杂度调整选项数量：

| 情境 | 选项数量 | 理由 |
|------|----------|------|
| 任务完成 | 2–3 | 建议后续工作流程步骤 |
| 用户询问 | 2–5 | 依问题复杂度调整 |
| 错误/失败 | 1–3 | 聚焦修复方向 |
| 进行中 | 1–2 | 继续或调整 |
| 信息回复 | 1–3 | 探索方向 |

**硬性上限**：不得超过 5 个选项。裁减至最相关的选项。

### 规则 5：优先使用斜杠命令

当下一步建议对应到已知的斜杠命令时，使用 `` `/command` `` 格式引用，让用户可以直接复制执行。没有对应命令的步骤使用自然语言描述。

## 情境模板

### 模板 1：任务完成

任务、技能执行或代码修改完成时使用。

```markdown
> **建议下一步：**
> - 执行 `/command1` 做 X
> - 执行 `/command2` 做 Y ⭐ **推荐** — [理由]
> - 执行 `/command3` 做 Z
```

### 模板 2：用户询问

AI 需要用户做出选择或提供信息时使用。

```markdown
> **请选择：**
> - **(A) 选项描述** — 补充说明
> - **(B) 选项描述** ⭐ **推荐** — [理由]
> - **(C) 选项描述** — 补充说明
```

### 模板 3：错误/失败

报告错误、失败或异常状态时使用。

```markdown
> **建议修复方向：**
> - 选项描述 ⭐ **推荐** — [理由]
> - 选项描述
```

### 模板 4：进行中

完成多步骤任务的中间阶段时使用。

```markdown
> **当前进度：[N/M]。下一步：**
> - 继续下一阶段 ⭐ **推荐**
> - 调整方向或参数
```

### 模板 5：信息回复

回答知识性问题或提供说明时使用。

```markdown
> **建议下一步：**
> - 深入了解 [相关主题]
> - 执行 `/command` 将知识应用到实现
```

## 格式规范

### 结构

所有导航区块使用 Markdown 引用区块（`>`）：

```markdown
> **[标题]：**
> - [选项/建议] ⭐ **推荐** — [理由]
> - [选项/建议] — [补充说明]
```

### 依情境的标题

| 情境 | 标题 |
|------|------|
| 任务完成 | `建议下一步` |
| 用户询问 | `请选择` |
| 错误/失败 | `建议修复方向` |
| 进行中 | `当前进度：[N/M]。下一步` |
| 信息回复 | `建议下一步` |

### 本地化

对于双语项目，导航区块遵循对话语言。本地化对照：

| English | 繁體中文 | 简体中文 |
|---------|----------|----------|
| Suggested next steps | 建議下一步 | 建议下一步 |
| Please choose | 請選擇 | 请选择 |
| Suggested resolution | 建議修復方向 | 建议修复方向 |
| Progress: [N/M]. Next | 目前進度：[N/M]。下一步 | 当前进度：[N/M]。下一步 |
| Recommended | 推薦 | 推荐 |

## 与技能的整合

### 现有技能

已有 `## Next Steps Guidance | 下一步引导` 区块的技能：

1. **保留**技能特有的场景化建议内容
2. **对齐**格式以符合本标准定义的模板
3. **加入**推荐标记（⭐）

### 新技能

新技能应包含遵循本标准模板的 `## Next Steps Guidance | 下一步引导` 区块。

### 超越技能

本标准适用于**所有 AI 响应**，不仅限于技能执行。一般对话、代码审查、调试过程及任何其他交互，在产生实质性响应时都必须遵循这些导航规则。

---

## 跨 AI 工具适用性

本标准与工具无关。适用于所有使用 UDS 标准的 AI 工具：

- Claude Code
- Cursor
- Windsurf
- GitHub Copilot
- Cline
- Roo Code
- Augment Code
- Trae
- 其他兼容工具

每个工具的集成层负责根据其 UI 能力渲染导航区块，但行为规则保持一致。

## 示例

### 示例 1：完成功能实现后

```markdown
> **建议下一步：**
> - 执行 `/test` 为新功能编写测试
> - 执行 `/commit` 提交变更 ⭐ **推荐** — 变更已完整且通过验证
> - 执行 `/review` 进行自我审查
```

### 示例 2：询问用户设计问题

```markdown
> **请选择：**
> - **(A) 使用新的数据库表** — 更灵活，迁移成本较高
> - **(B) 扩展现有表增加列** ⭐ **推荐** — 迁移更简单，满足当前需求
> - **(C) 使用 NoSQL 存储** — 适合非结构化数据但增加基础设施
```

### 示例 3：报告测试失败

```markdown
> **建议修复方向：**
> - 检查 `tests/auth.test.js:42` 的失败断言 ⭐ **推荐** — 错误信息指向空引用
> - 执行 `/debug` 进行系统性调试
> - 还原上次变更并调查
```

### 示例 4：工作流程中间进度更新

```markdown
> **当前进度：[2/5]。下一步：**
> - 继续第 3 阶段：实现 ⭐ **推荐**
> - 在继续前重新检视第 2 阶段需求
```

### 示例 5：回答技术问题

```markdown
> **建议下一步：**
> - 探索 `config/auth.yaml` 中的相关配置
> - 执行 `/sdd` 创建规格（若计划修改此行为）
```
