# 代码审查指南

**版本**: 1.0.0
**最后更新**: 2025-12-30
**适用范围**: 所有需要代码审查流程的项目

---

## 目的

本标准定义代码审查的最佳实践，确保代码质量、知识分享和团队协作。

---

## 审查清单

### 功能性

- [ ] 代码实现了预期功能
- [ ] 边界条件已处理
- [ ] 错误情况已处理
- [ ] 没有明显的逻辑错误

### 可读性

- [ ] 命名清晰有意义
- [ ] 代码结构合理
- [ ] 注释适当且有帮助
- [ ] 没有过度复杂的逻辑

### 安全性

- [ ] 输入已验证
- [ ] 没有硬编码的敏感信息
- [ ] SQL 注入已防范
- [ ] XSS 攻击已防范

### 性能

- [ ] 没有明显的性能问题
- [ ] 数据库查询已优化
- [ ] 没有不必要的循环
- [ ] 缓存使用适当

### 测试

- [ ] 有充分的测试覆盖
- [ ] 测试案例有意义
- [ ] 边界条件有测试
- [ ] 所有测试通过

---

## 审查者指南

### 提供建设性反馈

**好的反馈**:
```
建议：这个循环可以用 filter() 简化，
这样代码会更易读。例如：
items.filter(item => item.active)
```

**差的反馈**:
```
这代码太差了
```

### 区分必要和建议

- 🔴 **必要修改**: 安全问题、错误、不符合标准
- 🟡 **建议修改**: 代码风格、性能优化、可读性改进
- 🟢 **可选修改**: 个人偏好、微小改进

### 及时响应

| 优先级 | 响应时间 |
|-------|---------|
| 紧急 | 2 小时内 |
| 普通 | 1 个工作日内 |
| 低优先级 | 2 个工作日内 |

---

## 提交者指南

### 准备拉取请求

- [ ] 自己先审查一遍代码
- [ ] 确保所有测试通过
- [ ] 提供清晰的 PR 描述
- [ ] 关联相关 issue
- [ ] 保持 PR 规模适当（<400 行）

### 响应反馈

- 感谢审查者的时间
- 解释你的实现选择
- 开放接受建议
- 及时处理评论

## 快速参考卡

### 审查优先级

| 检查项 | 优先级 |
|-------|-------|
| 安全漏洞 | 🔴 最高 |
| 功能错误 | 🔴 高 |
| 测试覆盖 | 🟡 中 |
| 代码风格 | 🟢 低 |

### 评论前缀

| 前缀 | 含义 |
|------|------|
| `blocker:` | 必须修改 |
| `suggestion:` | 建议修改 |
| `question:` | 需要澄清 |
| `nit:` | 小问题 |
| `praise:` | 表扬好代码 |
