# 技术债管理标准

> 版本: 1.0.0 | 最后更新: 2026-04-01

## 概述

技术债代表因选择权宜之计而非更佳方案所产生的未来重工隐含成本。本标准定义如何分类、登记、排序、预算、衡量及追踪技术债。

## 1. 技术债类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **Design Debt** | 架构捷径或次优设计决策 | 缺少抽象层、紧耦合 |
| **Code Debt** | 增加维护负担的代码质量问题 | 重复代码、过长方法、Magic Number |
| **Test Debt** | 不足或低质量的测试覆盖 | 缺少单元测试、不稳定测试 |
| **Documentation Debt** | 缺失、过时或不正确的文档 | 未文档化的 API、过时 README |
| **Dependency Debt** | 过时、有漏洞或不必要的依赖 | 未修补的库、已弃用框架 |
| **Infrastructure Debt** | 次优的构建、部署或运维工具 | 手动部署、缺少监控 |

### 蓄意 vs 非蓄意

- **蓄意（Deliberate）**：有意识地选择捷径，应立即登记
- **非蓄意（Inadvertent）**：因知识不足而产生，发现后应立即登记

## 2. 技术债登记表

每个技术债项目必须记录以下 11 个字段：

| # | 字段 | 说明 |
|---|------|------|
| 1 | **ID** | 唯一标识符，格式 `TD-NNN` |
| 2 | **Title** | 简短描述性标题 |
| 3 | **Type** | 上述 6 种类型之一 |
| 4 | **Source** | 引入方式（蓄意/非蓄意） |
| 5 | **Impact** | 业务与技术影响描述 |
| 6 | **Estimated Cost** | 解决所需工作量 |
| 7 | **Interest** | 不解决的持续成本 |
| 8 | **Priority** | 优先等级（P0-P3） |
| 9 | **Owner** | 负责团队或个人 |
| 10 | **Created Date** | 登记日期 |
| 11 | **Target Resolution Date** | 计划解决日期 |

## 3. 预算分配

| 团队状态 | 预算 | 说明 |
|---------|------|------|
| 新项目（< 6 个月） | **10%** | 预防性维护 |
| 成熟项目（6+ 个月） | **15%** | 持续维护 |
| 高债务项目 | **20-30%** | 积极偿还 |

## 4. 优先排序矩阵

|  | 低工作量 | 中工作量 | 高工作量 |
|--|---------|---------|---------|
| **高影响** | P0 — 立即 | P1 — 下个 Sprint | P1 — 下个 Sprint |
| **中影响** | P1 — 下个 Sprint | P2 — 本季 | P2 — 本季 |
| **低影响** | P2 — 本季 | P3 — Backlog | P3 — Backlog |

### 利息类型

| 利息类型 | 说明 |
|---------|------|
| **时间利息** | 绕过债务所需的额外开发时间 |
| **风险利息** | Bug、宕机或安全事件的概率增加 |
| **人才利息** | 对开发者体验与留才的负面影响 |

## 5. 量化指标

| 指标 | 公式 | 目标 |
|------|------|------|
| 总债务量 | 所有未关闭债务项目数 | 下降趋势 |
| 债务比率 | 债务项目 / 总 Backlog × 100% | < 15% |
| 平均存活天数 | 总天数 / 债务项目数 | < 90 天 |
| 类型分布 | 各类型占比 | 单一类型不超过 40% |
| 高优先比率 | (P0+P1) / 总数 × 100% | < 20% |

## 6. Commit 标记

引入债务时：`Tech-Debt: TD-042 introduced`
解决债务时：`Tech-Debt: TD-042 resolved`
